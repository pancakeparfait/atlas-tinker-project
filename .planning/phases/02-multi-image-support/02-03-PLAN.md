Provide the client-side data layer for multi-image. After this plan, components in plans 04/05/06 can call typed hooks (useRecipeImages, useUploadRecipeImages, useDeleteRecipeImage, useReorderRecipeImages) without touching fetch directly, and cache invalidation keeps recipe detail and list views fresh.

Purpose: Centralize image network IO so UI components stay declarative. The Recipe interface gains two new fields (`images`, `primaryImageId`) so the list/detail components in plans 04 and 06 can read them off the existing useRecipe / useRecipes hooks.

Output: One new query module, one extended interface, one test file covering URL/body shape and invalidation behavior.

```typescript
export interface RecipeImageMetadata {
  id: string;
  order: number;
  fileName: string;
  mimeType: string;
}

export const recipeImageKeys = {
  all: (recipeId: string) => ['recipes', 'images', recipeId] as const,
};

export function useRecipeImages(recipeId: string): UseQueryResult<{ images: RecipeImageMetadata[] }, Error>;
export function useUploadRecipeImages(): UseMutationResult<{ imageIds: string[]; failed: Array<{ fileName: string; error: string }> }, Error, { recipeId: string; files: File[] }>;
export function useDeleteRecipeImage(): UseMutationResult<{ ok: true }, Error, { recipeId: string; imageId: string }>;
export function useReorderRecipeImages(): UseMutationResult<{ ok: true }, Error, { recipeId: string; orderedIds: string[] }>;
```
