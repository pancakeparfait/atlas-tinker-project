## Surfaces and Component Inventory

### Surface 1: Multi-Image Upload Zone (IMG-01, IMG-06)

Component: `MultiImageUploadZone`

**States:**

| State | Visual |
|-------|--------|
| Idle | Dashed border (`border-dashed border-2 border-border`), `--secondary` fill, upload icon (lucide `Upload`, 24px) centered, instruction text below icon |
| Drag-over | Solid border (`border-primary`), `--primary/10` tint fill, instruction text changes to "Drop to add" |
| Uploading | Per-file progress row: file name (14px/400) + progress bar (accent fill, 4px height, full width of row) + file size in muted |
| Upload complete | Progress bar replaced by green check (`lucide CheckCircle2`, 16px, accent color) |
| Upload error | Progress bar replaced by `lucide AlertCircle` (16px, destructive color) + inline error message |
| Disabled | 50% opacity, no pointer events |

**Layout:**
- Minimum height: 120px
- Border radius: `--radius` (0.5rem)
- Padding: 24px (lg)
- File input is visually hidden; zone is the click/drag target

**Validation feedback (IMG-06):**
- File too large (>10MB): "Photo must be under 10MB" shown inline below file name, destructive color
- Wrong file type: "Only JPEG, PNG, WebP, and GIF are supported" shown inline, destructive color
- Rejected files are shown in the error state row but do not block valid files in the same drop

### Surface 2: Image Gallery — Hero + Thumbnail Strip (IMG-02)

Component: `ImageGallery`

**Hero image:**
- Container: `position: relative`, `w-full`, `h-64 md:h-96` (256px mobile / 384px desktop)
- Border radius: `--radius`
- Image: `next/image` with `fill` + `object-cover`
- Loaded from: `/api/recipes/[id]/images/[imageId]`
- Clicking a thumbnail updates the displayed `imageId` in hero (client state, no page reload)

**Detail page gate fix (Pitfall 6):**
- The existing `recipe.imageUrl &&` gate at `page.tsx:159` MUST be replaced by `recipe.images && recipe.images.length > 0` check using the new images array. The `ImageGallery` component is the single source of truth for image display on the detail page.

**Thumbnail strip:**
- Layout: horizontal flex, `gap-2` (8px), `overflow-x-auto`, `py-2` (8px vertical padding)
- Thumbnail size: 80px × 80px fixed, `object-cover`, `rounded-sm`
- Active thumbnail (currently shown in hero): `ring-2 ring-primary ring-offset-2`
- Inactive thumbnail: no ring, 80% opacity, hover → 100% opacity (transition 150ms)
- Primary badge (order=0): absolute overlay bottom-left, "Primary" label, 12px/600, white text on `bg-primary`, `px-1 py-0.5`, `rounded-sm`
- Max thumbnails visible without scroll: 5–6 at typical viewport; strip scrolls horizontally on overflow

**Empty state (no images):**
- Container: same height as hero (`h-64 md:h-96`), `--secondary` fill, dashed border
- Icon: `lucide ImageOff`, 48px, muted-foreground
- Heading: "No photos yet" (20px/600)
- Body: "Add photos to help others follow along with ingredients, process, and the finished dish." (16px/400, muted-foreground, max-w-xs, centered)
- No CTA in empty state on detail page (upload is in edit view only)

### Surface 3: Sortable Thumbnail Strip in Edit View (IMG-04)

Component: `SortableThumbnailStrip` (wraps `@dnd-kit/sortable`)

**Drag affordance:**
- Each thumbnail has a `GripVertical` icon (lucide, 16px, muted-foreground) at top-right corner, visible always (not hover-only — ensures keyboard and touch discoverability)
- Entire thumbnail is draggable (listeners spread on container)
- Cursor: `grab` at rest, `grabbing` while dragging

**Drag states:**

| State | Visual |
|-------|--------|
| At rest | Normal thumbnail, grip icon visible |
| Drag in progress (active item) | 90% opacity, slight scale down (`scale-95`), `ring-2 ring-primary` |
| Drop target position | 2px solid primary-colored vertical gap opens between thumbnails to show insertion point |
| Drag complete | Item snaps to new position, transition 150ms ease-out |

**"Set Primary" affordance:**
- Reorder to first position automatically sets primary — this is the primary mechanism (locked in CONTEXT.md)
- No separate "Set as primary" button needed; first position crown badge makes this self-evident
- If user cannot drag (keyboard), Tab to thumbnail + Space/Enter opens a context menu with "Move to first (set primary)" option

**Delete per image (IMG-03):**
- Delete button: `lucide Trash2`, 16px, appears at top-right of thumbnail on hover/focus (overlaid, `bg-white/80` circle, `p-1`, `rounded-full`)
- On mobile (no hover): always visible at reduced opacity (70%), full opacity on focus
- Touch target: minimum 44px × 44px (achieved via `p-4` on the circle container hitbox)
- Keyboard: Tab to thumbnail, Tab again to reach delete button

### Surface 4: Delete Confirmation (IMG-03)

Pattern: `window.confirm` — matches existing recipe delete pattern (`page.tsx:27`). No modal component needed for Phase 2.

Confirmation copy:
- "Remove this photo from the recipe?" — OK to confirm, Cancel to abort
- Destructive action, no undo

### Surface 5: Recipe List Thumbnail (IMG-07)

Location: Recipe grid card (existing `Card` component)

**Thumbnail:**
- Size: 160px × 120px (aspect ratio ~4:3), `object-cover`
- Position: top of card, above title, `rounded-t-lg` to match card border radius
- Loaded from: `/api/recipes/[id]/images/[primaryImageId]` where `primaryImageId` is included in list API response

**Fallback (no primary image):**
- Placeholder: `--secondary` fill (`h-30`), `lucide Utensils` icon (32px, muted-foreground) centered
- No text label on placeholder — icon is sufficient at card size

**Skeleton (loading state):**
- `bg-muted animate-pulse` on the thumbnail area, same dimensions
- Matches existing TanStack Query loading patterns in the app

## Accessibility Notes

- Upload zone: `role="button"`, `aria-label="Upload recipe photos"`, `tabIndex={0}`, `onKeyDown` handles Enter/Space
- Drag-and-drop: keyboard alternative via Tab + Enter to open context menu ("Move to first") — @dnd-kit/sortable's `KeyboardSensor` handles arrow key reordering natively
- Thumbnails in gallery: `<img alt="Recipe photo N of M">` (numbered, since user-provided names are file names, not descriptions)
- Progress bars: `role="progressbar"` with `aria-valuenow` and `aria-valuemax`
- Delete buttons: `aria-label="Remove photo"` (no visible text label at thumbnail size)
- Primary badge: `aria-label="Primary photo"` on the overlay element

## Implementation Notes for Executor

1. **gallery single source of truth:** Replace `recipe.imageUrl &&` gate at `src/app/recipes/[id]/page.tsx:159` with `recipe.images && recipe.images.length > 0`. The `ImageGallery` component renders the hero; do not render a standalone `<Image>` alongside it.

2. **next/image sizing:** Hero image uses `fill` prop with `position: relative` parent. Thumbnails (80×80) use explicit `width={80} height={80}` props. List card thumbnails use explicit `width={160} height={120}` props.

3. **Upload progress UX:** Per RESEARCH.md Assumption A4, per-file progress is "N of M files done" counter, not byte-level. `fetch` is sufficient — no XMLHttpRequest needed.

4. **@dnd-kit API:** Use stable v1 API only — `@dnd-kit/core` + `@dnd-kit/sortable`. Import `useSortable` from `@dnd-kit/sortable`. Do not import from `@dnd-kit/react`.

5. **Thumbnail strip in edit vs. detail:** The sortable strip (with drag handles + delete) renders only in edit view. The read-only gallery (clickable to update hero, no drag handles) renders on detail view.

6. **Order normalization:** After delete, re-normalize remaining image orders in the same transaction before returning. UI should optimistically remove the thumbnail and re-number.
